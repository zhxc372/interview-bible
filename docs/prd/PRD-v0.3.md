> 基于 v0.2 双卡结构，新增 JD 入口模块。

---

# 0. 一句话定位

```text
用户输入一个 JD，系统输出一份可执行的面试备考地图。
```

---

# 1. 为什么需要 JD Intake

v0.2 的问题是：**用户不知道该学什么**。

知识点卡和面试卡很好用，但前提是用户已经知道"我要学哪个知识点"。实际上大多数人面对面试的第一步是：

```text
拿到一个 JD → 不知道该准备什么 → 要么乱学，要么焦虑
```

JD Intake 解决的就是这第一步：

```text
JD 输入 → JD 解构卡 → 备考科目图谱 → 知识点 Backlog → 逐个进卡片系统
```

---

# 2. 完整链路

```text
1. 输入 JD
   ↓
2. JD 解构卡
   - 岗位画像
   - 技术栈
   - 核心职责
   - 显性要求
   - 隐性要求
   - 风险点
   ↓
3. 备考科目图谱（12类分类法）
   - A. 编程语言
   - B. 数据结构与算法
   - C. 操作系统
   - D. 计算机网络
   - E. 数据库
   - F. 缓存/消息队列/中间件
   - G. 分布式系统
   - H. 系统设计
   - I. 工程实践
   - J. 项目经验
   - K. 排障与性能优化
   - L. 业务理解
   ↓
4. 知识点 Backlog
   - P0 必须会
   - P1 应该会
   - P2 加分项
   ↓
5. 项目证据映射
   - 哪些要求需要项目卡证明
   - 用户现有项目能覆盖哪些
   - 证据缺口
   ↓
6. 卡片生成队列
   - 哪些先做知识点卡
   - 哪些直接做面试卡
   - 哪些需要先补项目证据
   ↓
7. 下一步唯一动作
```

# 3. 面试科目分类法（12类）

```text
A. 编程语言
B. 数据结构与算法
C. 操作系统
D. 计算机网络
E. 数据库
F. 缓存/消息队列/中间件
G. 分布式系统
H. 系统设计
I. 工程实践
J. 项目经验
K. 排障与性能优化
L. 业务理解
```

每个 JD 都映射到这 12 类里。分类稳定，不会每次乱。

# 4. JD 解构不是简单拆关键词

JD 里的很多内容不是知识点，而是**能力信号**。

比如 JD 写：

> 熟悉 Go 并发编程，有高并发服务治理经验。

不能只拆成 goroutine、channel、mutex、context、GMP。

还要拆出：

| 类型 | 内容 |
|------|------|
| 知识点 | goroutine、channel、context、锁、调度器 |
| 工程能力 | 并发安全、超时控制、限流、熔断、排查 goroutine 泄漏 |
| 项目证据 | 是否做过并发任务调度、接口压测、性能优化 |
| 面试表达 | 如何讲清楚"我怎么设计并发模型" |
| 压力追问 | "你怎么证明不是纸上谈兵？" |

所以 JD 解构层必须存在。

---

# 5. 双线并行：知识线 + 证据线

```text
JD 解构
  ├── 知识线：科目 → 知识点卡 → 面试卡
  └── 证据线：项目要求 → 项目卡 → 压力追问
```

面试不是考试，面试是"知识 + 证据"的混合验证。
只会讲知识点 → 像背八股。
只讲项目 → 暴露基础不扎实。
两个并行。

- 能从 JD 中提取岗位画像
- 能区分显性要求和隐性要求
- 能生成备考科目图谱（映射到12类）
- 能生成 P0/P1/P2 知识点 Backlog
- 能指出哪些要求需要项目证据支撑
- 能给出下一步唯一动作
