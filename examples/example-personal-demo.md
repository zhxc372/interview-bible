## 示例输出

# 项目表达卡：Redis 幂等提交 Demo

项目类型：【个人实验】

## 1. 核心矛盾 / 技术难点
重复点击、网络重试或客户端重复提交时，同一个业务请求可能被处理多次。这个 Demo 重点验证如何用 requestId 做幂等保护。

## 2. 我的角色
我负责：Demo 设计、Redis key 设计、幂等判断逻辑、测试日志记录。  
我不负责：生产灰度、真实流量压测、分布式事务一致性、UI 完整交互。

## 3. 关键取舍
我选择：用 Redis 存 requestId，并设置过期时间。  
我放弃：只依赖数据库唯一索引或纯前端按钮禁用。  
原因：Redis 适合作为快速幂等门禁，但需要处理过期时间、失败重试和一致性边界。

## 4. 证据锚点
代码：有，本地代码片段。  
日志：有，重复提交测试日志。  
PR / Commit：缺失。  
压测：缺失。  
截图：缺失。  
Demo：有，本地可复现。  
文档：README。

## 5. 证据强度
强证据：代码片段、测试日志、README、本地可复现 Demo。  
弱证据：没有真实并发流量数据。  
缺失证据：没有生产环境指标，不能声称“支撑高并发线上系统”。

## 6. 压力追问
1. 为什么不用数据库唯一索引？
   - 风险点：需要讲清 Redis 门禁和 DB 约束的取舍，而不是简单说 Redis 快。
2. Redis key 过期时间怎么设？
   - 风险点：过短可能重复处理，过长可能占用内存或误拦截。
3. 如果业务处理成功但 Redis 写失败怎么办？
   - 风险点：需要暴露一致性边界，不能把 Demo 说成完整生产方案。

## 诚实表达建议
这可以表述为“我做过一个本地可复现的幂等提交实验，用来验证 requestId + Redis 的基本思路”，不要表述为“我在线上高并发系统中主导了幂等架构”。
